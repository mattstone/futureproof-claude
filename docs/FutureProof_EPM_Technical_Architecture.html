EPM Overview

Model structure and stakeholder relationships

FutureProof's Income from Property Equity solution is a fund-based income product that transforms home equity into sustainable monthly income. The model connects four stakeholders through a sophisticated distribution system powered by monthly rebalancing and performance monitoring.

The Four Stakeholders

Stakeholder Role Capital Flow
Customer Homeowner who partners equity for monthly income Receives monthly distributions from fund returns
Investment Partner Manages invested capital Generates returns; manages fund performance and rebalancing
Lender Provides mortgage product & customer relationship Receives margin on monthly distributions (not traditional interest)
Wholesale Funder Supplies capital to lenders via funding lines Receives wholesale margin on deployed capital

Monthly Distribution Flow

Fund Returns: Investment Partner's holdings generate returns that are aggregated and tracked monthly (on average 10% p.a. over 30 years).

Monthly Distribution: System calculates monthly payout amounts based on:

  • Customer's equity percentage (typically 2% of home value)
  • Fund performance month-to-date
  • Interest holiday status (if triggered)

Amount to Customer: Customer receives their share of monthly returns as income.

Lender Margin: Lender deducts their margin from distribution (0.75% to 2% depending on tier).

Wholesale Margin: Wholesale Funder receives their cut (0.2% to 3.6% depending on structure).

FutureProof Margin: Platform takes its cut (0.25%) for ongoing management.

Customer Journey

Three phases from origination to exit

Three Phases

PHASE 1: ORIGINATION (Weeks 1–4)

Customer applies: Provides property details, equity percentage, income expectations through portal.

Deal Agent processes: KYC/AML checks, property valuation, risk tier assignment, document generation.

e-Signature: Customer reviews and signs loan documents, investment agreement.

Funding confirmation: Lender & Wholesale Funder approve. Capital transferred. Account activated.

PHASE 2: ONGOING SERVICING (Months 1–30+)

Monthly distributions: Distribution Agent calculates returns every month. Customer receives payment on fixed date.

Performance monitoring: Investment Agent tracks fund performance. Generates rolling metrics visible on customer dashboard.

Interest holidays: If performance drops, holiday flag automatically triggered. Holiday status visible on customer statements and dashboard.

Statements & reports: Monthly statements show distribution amount, margins paid, fund performance, and holiday status. Accessible via portal.

Account management: Customer can request early exit, modify settings. Lender reviews requests within 5 business days.

PHASE 3: EXIT (Early or Maturity)

Customer-initiated: Request early repayment. Deal Agent calculates exit position (fund value minus outstanding fees).

Lender approval: Lender reviews exit request, confirms repayment terms.

Settlement: Distribution Agent processes final payment. All accounts reconciled. Loan closed.

Documents archived: All transaction history, statements, audit logs preserved for 7 years. Compliance Agent completes final audit report.

System Architecture

Platform components and core services

Core Platform

Rails 8.1 + Hotwire + Stimulus: Single monolithic Rails application with real-time updates via Hotwire (Turbo) for seamless user interactions and Stimulus controllers for client-side interactivity. Hotwire enables live dashboard updates (fund performance, distribution status, holiday triggers) without requiring WebSocket complexity—server-side changes automatically broadcast to clients via Turbo Streams.

PostgreSQL Database: Single source of truth for all customer, transaction, and account data with full audit trails. Uses JSON columns for flexible fund metadata storage, materialized views for performance metrics aggregation, and database triggers to enforce compliance rules automatically. Partitioned by customer ID for horizontal scaling.

Kafka Event Queue: Asynchronous event streaming for agent coordination and system-wide notifications. Agents subscribe to topics: distributions.*, holidays.*, compliance.*, deals.*. Ensures exactly-once processing with idempotent subscriber logic; dead-letter queues capture processing failures for manual remediation.

Redis Cache: Session management, rate limiting, and temporary computation caching. Used for: customer session state, monthly distribution calculations (staged before final commit), Bloom filters for fraud detection (quick checking of known patterns), and cache-aside pattern for frequently accessed data (fund performance metrics, holiday thresholds).

S3 Storage: Document archival, statements, and audit logs with versioning enabled. Uses server-side encryption (SSE-S3). Lifecycle policies archive logs after 90 days to Glacier, delete after 7 years (compliance requirement).

Core Services

Distribution Engine: Microservice (Ruby + Sidekiq) that runs on 1st-5th of month. Pulls monthly fund returns from Investment Partner, calculates per-customer distributions using formula: customer_equity_pct × fund_return × (1 - lender_margin) × (1 - funder_margin) × (1 - fp_margin). Stages calculations in Redis for audit, executes ACH/bank transfers (via Stripe), publishes distribution.executed event. Implements retry logic with exponential backoff for payment failures; failed payments trigger lender alerts for manual intervention.

Compliance Engine: Real-time service that validates every transaction against 20+ regulatory rules (KYC/AML, responsible lending, distribution accuracy, fund audits, data privacy). Writes audit trail to PostgreSQL with immutable event log table. Implements Change Data Capture (CDC) via database triggers—any mutation to customer/transaction tables triggers audit record creation. Violations escalate via Kafka topic compliance.violation to ops dashboard.

Fraud Detection: Machine learning pipeline (offline) trained on historical transaction patterns. Real-time scorer deployed as REST service checks each transaction/login against: (1) known fraud patterns (Bloom filter in Redis), (2) behavioral anomalies (isolation forest model), (3) rule-based checks (impossible geography, velocity limits). Assigns fraud score 0-100; scores >70 trigger manual review, >90 block transaction automatically.

Interest Holiday Logic: Scheduled job runs monthly (after distributions finalize). Queries fund performance history (stored in materialized view), calculates rolling 12-month returns, compares against entry threshold (0.90) and exit threshold (1.458). State machine manages transitions: normal → holiday_pending → holiday_active → recovery_in_progress → normal. Publishes state change events for customer notification.

Notification Service: Async job queue (Sidekiq) with multiple channels: email (SendGrid), SMS (Twilio), in-app notifications (database + Hotwire broadcast). Implements delivery guarantees: stores pending notifications in database with idempotent keys, retries failed sends with backoff, marks as delivered only after provider confirmation. Respects customer notification preferences (stored as JSON array in user record).

Document Engine: Server-side rendered (ERB templates) + Prawn gem for PDF generation. Stores templates in database for versioning/audit. Generates: loan documents (auto-filled with customer data), investment agreements (region-specific legal terms), monthly statements (tabular layout with performance charts), disclosures. All outputs digitally signed via AWS Signer service, archived to S3 with immutable version IDs.

Agent Technology Implementation

Framework: Four agents implemented as Rails service classes (e.g., DealAgent::OnboardingService, DistributionAgent::ProcessingService). Each agent runs as a Sidekiq background job (worker pool, 4 workers per pod). Agents are horizontally scalable—add more Sidekiq pods to increase throughput.

Event Subscription (Kafka): Each agent subscribes to relevant Kafka topics using ruby-kafka gem. Consumer groups enable automatic partitioning—if 4 agent pods are running, 4 consume partitions in parallel, automatically rebalancing if a pod dies. Dead-letter queues capture processing failures (poison pill messages, timeouts); operations team investigates manually.

Idempotency & Retry Logic: Agents implement idempotent handlers—processing same event twice produces same result (no duplicate charges, no double-distributions). Uses database unique constraints + transactional locks. Failed agent tasks auto-retry with exponential backoff (1s → 2s → 4s → 8s → 16s), then escalate to dead-letter queue.

Distributed Tracing: Every event gets trace_id (UUID) from source. As event flows through agents (Deal Agent → Compliance Agent → Distribution Agent), each writes logs tagged with trace_id. Enables end-to-end visibility: search CloudWatch by trace_id to see entire customer journey (from application submit → KYC check → funds release → first distribution).

State Management: Deal workflow uses state machine (AASM gem): draft → submitted → kyc_pending → approved → funded → active. Holiday logic uses similar state machine: normal → holiday_pending → holiday_active → recovery_in_progress → normal. State transitions trigger events; dependent agents react. Example: when Deal Agent transitions deal to approved, publishes deal.approved event; Distribution Agent subscribes and prepares to execute initial payment.

Call Center Integration (Twilio Flex): When human escalation needed (payment failure, KYC issue, customer calls), ops team receives ticket in Twilio Flex dashboard. Agent clicks customer, Flex dials phone via Twilio, logs call recording automatically. Call notes stored in customer account (alongside audit trail). If customer calls in (1-800-FUTUREPROOF), Flex IVR routes to available agent. Agent sees customer profile, recent interactions, account balance. Customer support ticket closed when agent resolves issue (update bank account, explain holiday, etc.).

Synchronous vs. Asynchronous: Agent actions default to async (publish event, job queued, returns immediately). Synchronous calls used only when timeout is critical and timeout is short: (a) Stripe payment processing (5-second timeout), (b) IDology KYC check (10-second timeout), (c) property valuation lookup (15-second timeout). All synchronous calls wrapped in circuit breaker pattern—if 3rd party times out, fallback to manual review (escalate to human).

Agent Communication Flow

All agents publish events to Kafka; others subscribe to relevant events and react asynchronously. This decouples the four workflows and enables real-time coordination without tight coupling.

Deal Agent publishes deal.created when customer onboarded. Distribution Agent and Compliance Agent subscribe to initiate workflows.

Distribution Agent publishes distribution.monthly on cutoff date. Lender and Wholesale Funder receive payments.

Investment Agent publishes holiday.triggered or holiday.released. Distribution Agent adjusts payment calculations.

Compliance Agent publishes fraud.detected or rule.violation. System escalates to lender/admin for review.

Agent Autonomy & Human Interaction Points

Fully Autonomous (No Human Needed):

  • Deal Agent: Customer applies through portal → KYC/AML automated checks via third-party (IDology) → Property valuation via Zillow API → Risk tier auto-assigned based on LTV ratio → Loan documents auto-generated and e-signed (DocuSign) → Funds released automatically once both parties signed. Human only needed if: KYC fails (flagged for manual review), property value can't be determined (research property details manually), customer disputes risk tier (escalation to lender).
  • Distribution Agent: Monthly cutoff date arrives → Pulls fund performance data from Investment Partner → Calculates customer distributions (deterministic formula) → Stages payments in Redis → Executes ACH transfers to customer bank accounts (via Stripe) → Generates statements and archives to S3. Human only needed if: Payment fails 3 times (retry backoff exhausted, manual bank reconciliation required), margin discrepancy detected by Compliance Agent (audit investigation), customer bank account invalid (call customer to update).
  • Investment Agent: Nightly, queries fund performance metrics → Calculates rolling 12-month returns → Compares to thresholds (0.90 entry, 1.458 exit) → Updates holiday flags in database → Publishes holiday.triggered or holiday.released events → Lender dashboard updates in real-time via Hotwire. Human only needed if: Fund data missing/delayed from Investment Partner (escalate to partner), threshold breach looks suspicious (audit historical data), customer disputes holiday status (support team explains via call center).
  • Compliance Agent: Every transaction triggers real-time compliance check → Fraud score computed (ML model) → Rule violations flagged automatically → Suspicious activities escalated to dashboard (red alert). Human only needed if: High-confidence fraud detected (>90 score, block + investigate), potential regulatory violation found (escalate to legal), audit findings need remediation (remediation plan + implementation).

Required Human Interactions (Escalation Points):

  • Customer Support: Customers call/chat when: account questions (balance, distribution amount), payment failed (reroute to different bank account), early exit request (requires lender approval), fraud alert on account (verification call), holiday status confusion (explain via call center agent).
  • Lender Review & Approval: Lender dashboard shows: pending loan applications (KYC failed), customers requesting early exit, high-risk accounts flagged by Compliance Agent, distribution payment failures, holiday status changes for their portfolio. Lender clicks "Approve" or "Reject"—system executes next step autonomously.
  • Wholesale Funder Operations: Funder dashboard tracks: portfolio health metrics (yield, default rate), new lender tier requests, funding line increases, concentration risk alerts. Funder approves funding requests; system automatically updates lender credit lines in database.
  • Investment Partner Reconciliation: Investment Partner delivers monthly fund performance data. System auto-validates against Compliance Agent rules (reconciliation job compares reported returns vs. ETF market data). If mismatch >0.1%, triggers manual audit (query Investment Partner, investigate data quality).
  • Ops & Monitoring: Dashboard alerts: failed ACH transfers (manual bank reconciliation), high fraud scores requiring review, Kafka lag >1000 messages (investigate agent slowdown), Sentry errors (developer investigation), monthly compliance reporting (internal audit team reviews logs).

Example: Autonomous Distribution Month

Day 1 (Cutoff Date, 9 AM Sydney):

  1. Distribution Agent wakes (scheduled job via Sidekiq)
  2. Queries accounts table: 5,000 customers with deleted_at IS NULL
  3. Pulls fund_performance materialized view for this month
  4. For each account: calculates customer_payout (formula: fund_return × (1 - margins) × holiday_adjustment)
  5. Stages all 5,000 distributions in Redis transaction (atomic)
  6. Publishes distribution.stage_complete event
  7. Compliance Agent subscribes: validates 5,000 distributions against rules (no negative payouts, margins correct, no duplicate processing)
  8. If all pass: Distribution Agent commits to database (INSERT into distributions table, immutable)
  9. No human needed yet — fully automated

Day 2-3 (Payment Execution, 6 PM):

  1. Distribution Agent wakes for payment batch
  2. Queries distributions table WHERE payment_status = 'staged'
  3. Initiates Stripe ACH transfers for all 5,000 customers (batch mode, lower fees)
  4. Stripe returns charge IDs and status
  5. Updates payments table with stripe_charge_id and status
  6. 4,950 succeed, 50 fail (invalid account, insufficient funds, account closed)
  7. Distribution Agent publishes distribution.payment_partial_failure event
  8. HUMAN ESCALATION TRIGGERED: Alerts go to: (a) Lender dashboard (red alert: "50 payment failures for your customers"), (b) Ops team via PagerDuty, (c) Customer support queue (create tickets for affected 50 customers)
  9. Lender manually reviews failed customers, contacts them via call center to update bank details
  10. Customer updates account via self-service portal or calls support
  11. Distribution Agent retries failed payments next day (automatically, no human action needed)

Key Principle: Agents execute deterministic workflows autonomously. Human interaction is minimal and targeted—only when exceptions occur, approvals are needed, or customers initiate actions. This keeps operational costs low while maintaining compliance oversight.

Interest Holiday Mechanism

Dynamic payment suspension based on fund performance

How Holidays Work

Interest holidays provide borrowers protection during market downturns. The Investment Agent monitors rolling fund performance against dynamic thresholds on a monthly basis.

Parameter Value Meaning
Entry Threshold 0.90 (90%) If fund value drops to 90% of starting value, entry conditions met
Exit Threshold 1.458 (145.8%) When fund recovers to 145.8% of original value, holiday automatically released
Maximum Duration Variable Holiday automatically expires if conditions not met within timeframe
Accumulated Interest Ongoing During holiday, interest is not distributed to customer but accumulates in the account

Timeline Example

Months 1–10: Normal Operations

Fund performing well. Customer receives monthly distributions. Investment Agent monitors thresholds monthly.

Month 11: Market Downturn

Fund value drops to 88% of starting value. Entry threshold (90%) breached. Investment Agent flags interest holiday status in customer account.

Months 12–18: Holiday Flagged

Interest holiday flag remains active. Customer continues receiving monthly distributions. Lender monitoring alert active. Monthly statements indicate holiday status.

Month 19: Recovery Confirmed

Fund recovers to 145.8% threshold. Investment Agent removes holiday flag. Status returns to normal operations.

Real-Time Fraud Monitoring

Account Access Anomalies: Logins from unusual locations, rapid device changes, after-hours access patterns.

Unusual Transaction Patterns: Multiple withdrawals same day, requests immediately after distribution, systematic holiday manipulation.

Data Inconsistencies: Mismatches between reported and actual fund values, distribution calculation errors, lender margin discrepancies.

Regulatory Violations: KYC/AML failures, undisclosed conflicts of interest, unauthorized fund transfers.

Alerts are automatically escalated to lenders and funders. Critical incidents require manual review.

Audit & Reporting

Audit Trail: Every transaction, margin payment, and holiday trigger is recorded with timestamp, user ID, and change details. Logs retained for 7 years minimum (compliance requirement).

Monthly Reporting: Lenders and wholesale funders receive automated reports on portfolio health, margin collections, and default scenarios.

Annual Compliance Certification: To be performed by external auditor. Covers all regulatory requirements across Australia, UK, and US jurisdictions.

Architecture Notes

Monolithic Rails App: Single codebase simplifies deployment, testing, and debugging. Uses Rails concerns for code organization (e.g., Distributable concern shared across models). Real-time features powered by Hotwire (Turbo Streams) enable dashboard updates without JavaScript complexity—when a holiday is triggered, server broadcasts HTML fragment to connected clients. Health checks (Rails endpoint returning JSON) monitored by Fly.io load balancer; failed instances auto-replaced.

Event-Driven Agents: Agents implemented as background jobs (Sidekiq with 4 worker processes per pod) listening to Kafka topics. Each agent is a Rails service class (e.g., DealAgent::OnboardingService) with single responsibility. Kafka Consumer Group ensures horizontal scalability—multiple agent instances process partitions in parallel. Implements distributed tracing via OpenTelemetry; each event tagged with trace ID for end-to-end visibility.

PostgreSQL as Single Source of Truth: All customer, transaction, and account data stored in single database with logical replication enabled (standby in secondary region for disaster recovery). Audit trail implemented as immutable event log—no UPDATE/DELETE allowed, only INSERT. Uses SERIAL ID for ordering, JSON data for flexible schema evolution. Connection pooling (PgBouncer) limits database load; idle connections recycled after 30 minutes.

Caching Strategy: Three-tier: (1) Database query cache (Rails fragment caching for dashboard components, 1-hour TTL), (2) Redis session cache (customer login state, 7-day rolling expiry), (3) Application-level cache (monthly metrics computed once, stored in Redis, invalidated on new distribution). Implements cache-aside pattern with lazy loading.

Key Design Patterns

Immutable Logs: distributions, payments, and audit_log tables have no UPDATE/DELETE permissions. All changes recorded as INSERT-only events with timestamps. Ensures regulatory compliance and audit trail integrity.

Soft Deletes: customers and accounts include deleted_at column (nullable). Logical deletion preserves historical data; queries filter WHERE deleted_at IS NULL. Allows account reinstatement without data loss.

JSON Columns: audit_log.changes_json stores before/after values (flexible schema). fraud_alerts.details_json stores detection model output, thresholds, and matching rules. accounts.metadata_json stores region-specific terms, holiday parameters, customer preferences.

Materialized Views: fund_performance refreshed nightly (expensive aggregation of historical returns). Queries use view instead of computing on-the-fly. Improves dashboard load times (sub-100ms queries). monthly_margins_summary pre-aggregates margins by lender/funder for reporting.

Partitioning: distributions and payments tables partitioned by created_at (monthly partitions). Improves query performance for range scans (e.g., "all distributions in March"). Partitions older than 1 year moved to cold storage (S3).

Indexes: Composite indexes on common queries: (customer_id, created_at) for customer history lookups, (account_id, distribution_month) for account statements, (status, created_at) for failed payment recovery. Foreign key indexes auto-created on customer_id, account_id, lender_id.

Example: Monthly Distribution Query

Scenario: Calculate customer's March 2026 distribution.

SELECT
  a.customer_id,
  a.current_balance,
  fp.return_pct,
  (a.current_balance * fp.return_pct) AS gross_return,
  (a.current_balance * fp.return_pct * (1 - l.margin_rate) * (1 - wf.margin_rate) * 0.9975) AS customer_payout
FROM accounts a
JOIN fund_performance fp ON a.id = fp.account_id AND fp.month = '2026-03'
JOIN lenders l ON a.lender_id = l.id
JOIN wholesale_funders wf ON l.funder_id = wf.id
WHERE a.customer_id = $1 AND a.deleted_at IS NULL;

Result fed to Distribution Agent for payment execution. All calculations recorded in audit_log for compliance.

Deployment & Quality Assurance

Release process and testing strategy

Testing Strategy

Unit Tests: Core business logic (distribution calculations, holiday logic, fraud detection rules).

Integration Tests: Agent workflows, end-to-end customer journeys, payment processing.

Performance Tests: Monthly distribution batch processing at scale (1000s of customers), query optimization.

Security Tests: Authentication, authorization, data access controls, PII encryption.

Manual Testing: Customer portal UX, lender reporting accuracy, admin workflows.

Continuous Deployment

GitHub Actions: Automated tests run on every PR: RSpec (unit/integration, 200+ specs), Rubocop (linting), Brakeman (security), database migrations validated. Tests must pass + code review approval before merge to main. Deploys require second approval from ops team.

Staging Environment: Deployed automatically from main branch using Fly.io's staging app (same infra as production, 1 instance instead of 3). Runs against production-replica database (encrypted, daily sanitized PII removal). Manual smoke tests performed: login, distribution calculation, payment processing, fraud alert generation. Staging data refreshed nightly from production (read-only copy).

Production Release: Blue-green deployment on Fly.io. Traffic gradually shifted 10% → 50% → 100% to new version over 10 minutes (canary rollout). Health checks verify: (1) Rails boot successful, (2) database connection working, (3) Kafka topics reachable, (4) core endpoints respond with <200ms latency. Automatic rollback if health checks fail within 5 minutes or if error rate spikes >2% above baseline.

Database Migrations: Zero-downtime migrations using strong_migrations gem (prevents destructive migrations). Tests run against production-replica before execution. Deployments include pre-deployment (create columns, indexes) and post-deployment (backfill data, drop old columns) steps with manual approval gates.

Monitoring & Alerting

Sentry: Real-time error tracking with source map support (debuggable minified JavaScript). Groups errors by fingerprint, alerting on new/regressed errors. Critical errors (distribution failures, fraud detection timeouts) trigger PagerDuty escalations (→ on-call engineer) within 2 minutes of occurrence.

CloudWatch Logs: Structured JSON logging (via Rails lograge gem). Each request logs: trace_id, user_id, duration_ms, database_queries, external_api_calls. Stores in CloudWatch with retention policy (90 days hot, archival to S3 after 90 days). Custom metrics: distribution execution time, payment success rate, fraud detection latency.

Dashboard (Grafana): Real-time visibility into: (1) System health (CPU, memory, disk usage per pod), (2) Application metrics (requests/sec, error rate, p95 latency), (3) Business metrics (distributions executed, total payments, customer count, revenue per month), (4) Agent health (processing lag per Kafka topic, dead-letter queue size), (5) Database health (connection pool usage, replication lag). Alerts fire when: error rate >1%, distribution latency >60s, Kafka lag >1000 messages, database replication lag >30s.
